Key Invariants

1. One lane per agent, concurrency=1 — no overlapping invocations
2. Cursor advances only after processing — crash-safe message handling
3. Zero LLM cost when inbox empty — ticks are free if nothing to do
4. Super-user messages always ACT_NOW — bypass triage classification
5. Agents never call ack — daemon owns cursor management (HIVE_DAEMON_SPAWN=1)
6. Staggered ticks — prevents thundering herd of concurrent API calls
7. Memory written before read — NOTE/QUEUE saved to disk before marking read

People Table

CREATE TABLE people (
  id        INTEGER PRIMARY KEY,
  alias     TEXT UNIQUE NOT NULL,    -- e.g. "ceo", "eng-mgr"
  name      TEXT NOT NULL,
  roleTemplate TEXT,                 -- "ceo" | "manager" | "swe" | "qa"
  status    TEXT DEFAULT 'active',
  folder    TEXT NOT NULL,           -- relative path under org/
  reportsTo INTEGER REFERENCES people(id),
  createdAt DATETIME DEFAULT CURRENT_TIMESTAMP
);

Person ID 0 = super-user (human). Hierarchy defined by reportsTo, not directory nesting.

Org Boundary Rules

  • CEO: Delegates to direct reports only. Never skip-level assigns.
  • Managers: Delegate to direct reports. Never do IC work themselves.
  • ICs: Execute work, report back to manager, request peer reviews.

Agents

What is an Agent

A Claude CLI process spawned by the daemon. Stateless — each invocation gets a fresh session with an assembled system prompt.

Spawner

spawnClaude(args, { cwd: agentDir, input: workContext, timeoutMs, env })

CLI args

-p --model <model> --system-prompt <prompt> --permission-mode bypassPermissions --output-format json

Environment

VarValuePurpose
GIT_AUTHOR_NAME"Name (hive/alias)"Agents sign commits
GIT_AUTHOR_EMAIL"alias@hive.local"Git identity
HIVE_DAEMON_SPAWN"1"Blocks ack in CLI (prevents cursor corruption)
CLAUDECODEdeletedAllows spawning from within Claude Code

Messages

CREATE TABLE messages (
  seq            INTEGER,              -- per-conversation sequential
  conversationId TEXT,
  senderId       INTEGER,
  content        TEXT,
  timestamp      DATETIME DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (conversationId, seq)
);

Sent atomically: BEGIN → SELECT MAX(seq) → INSERT (max+1) → COMMIT

Read Cursors

CREATE TABLE cursors (
  personId       INTEGER,
  conversationId TEXT,
  lastSeq        INTEGER DEFAULT 0,     -- never goes backward
  updatedAt      DATETIME DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (personId, conversationId)
);

Unread: messages WHERE seq > cursor AND senderId != personId

Cursor Corruption Prevention

  • Agent subprocesses have HIVE_DAEMON_SPAWN=1 → hive chat ack is a no-op
  • Only the daemon advances cursors after processing through checkWork
  • Audit logging on every cursor move with warning if cursor jumps ahead of max seq

Chat Adapter

Bridge between daemon (aliases) and chat store (person IDs):

postMessage(senderAlias, conversationId, content) → syntheticId
getUnread(alias) → UnreadMessage[]
markRead(alias, messageIds) → void

Access Control

  • DM: Both participants can read/write. Super-user can DM anyone.
  • Group: Only members can read/write.

Daemon

Startup Sequence

graph TB START["daemon.start()"] --> PID["Write PID file"] PID --> REG["Register agents in state store"] REG --> CRASH["Crash recovery
(recover stale 'working' agents)"] CRASH --> FUP["Init FollowUpStore + Scheduler"] FUP --> MEM["Index memories (background)"] MEM --> TICK["Schedule staggered ticks"]

Tick Staggering

tickMs = 600_000  (10 min)
staggerMs = tickMs / agentCount

Agent 0: fires at t=0,          repeats every 10min
Agent 1: fires at t=stagger,    repeats every 10min
Agent 2: fires at t=2*stagger,  repeats every 10min
...

Prevents thundering herd of concurrent Claude CLI spawns hitting API rate limits.

Lanes

Per-agent FIFO queue. Concurrency = 1.

  • Serial per-agent: No overlapping invocations for same agent
  • Parallel cross-agent: Different agents run independently
  • drain(): Wait for all pending tasks (graceful shutdown)

Signal Path (fast response)

message posted → POST /api/signal → daemon.signalConversation(id)
  → resolve members → for each agent member:
    → debounce 100ms → enqueueCheckWork(agentId)

CheckWork Cycle

graph TB CW["checkWork(agent)"] --> GUARD{"status == working?"} GUARD -->|"yes"| SKIP["Skip"] GUARD -->|"no"| READ["Read unread messages"] READ --> EMPTY{"inbox empty?"} EMPTY -->|"yes"| DONE["Return (zero cost)"] EMPTY -->|"no"| SCORE["Stage 1: Score"] SCORE --> TRIAGE["Stage 2: Triage (haiku, 5min timeout)"] TRIAGE --> CLASSIFY{"classification?"} CLASSIFY -->|"NOTE/QUEUE"| MEMORY["Append to memory file"] MEMORY --> MARKREAD["Mark read"] CLASSIFY -->|"IGNORE"| MARKREAD CLASSIFY -->|"ACT_NOW"| WORKING["Set status=working"] WORKING --> PROMPT["Assemble prompt + memory search"] PROMPT --> SPAWN["Spawn Claude CLI"] SPAWN --> PARSE["Parse FOLLOWUP tags"] PARSE --> AUDITLOG["Log to audit"] AUDITLOG --> IDLE["Set status=idle"] IDLE --> RECHECK{"recheck?"} RECHECK -->|"yes"| CW

Exponential Decay

When inbox has messages but none ACT_NOW: schedule recheck with delays [30s, 60s, 180s]. Resets when inbox empties.

Crash Recovery

  1. On start: scan agent_state for status='working'
  2. Check if PID alive: process.kill(pid, 0)
  3. Dead PID → mark idle (recovered)
  4. Rate limit: 3+ crashes in 10min → status=errored, skip ticks

Shutdown

daemon.stop():
  running = false → clear tick timers → stop followup scheduler
  → drain all lanes → remove PID file

Triage Pipeline

Stage 1: Deterministic Scoring

ComponentWeightLogic
Authority0.25Super-user=10, Manager=10, Report=3, Peer=5, Unknown=1
Urgency0.25metadata.urgent=10, else 0
Conversation0.20DM=8, Group=5
Recency0.15Linear decay: 0h=10, 24h=0
Mention0.15@agent in mentions=10, else 0

Formula: score = Σ(component × weight), clamped [0, 10]

Stage 2: LLM Classification

ClassTriggerAction
ACT_NOWManager request, urgent, @mention with questionSpawn full agent
QUEUENew task, non-urgent request, FYI needing follow-upAppend to memory, mark read
NOTEAnnouncements, status updates, useful contextAppend to memory, mark read
IGNOREOther teams, social chatterMark read, drop

Override: Super-user messages → always ACT_NOW

Timeout: 5 minutes. On failure: default all to QUEUE

Tag Format

FOLLOWUP: <description>
| check: <shell command>    # exit 0=done, 1=not done, 2=skip
| backoff: 10m, 30m, 1h     # intervals between checks

Storage (orchestrator.db)

CREATE TABLE followups (
  id               INTEGER PRIMARY KEY AUTOINCREMENT,
  agent_id         TEXT NOT NULL,
  description      TEXT NOT NULL,
  check_command    TEXT,
  backoff_schedule TEXT NOT NULL,     -- JSON array of ms values
  attempt          INTEGER DEFAULT 0,
  next_check_at    DATETIME NOT NULL,
  last_check_exit  INTEGER,
  last_check_output TEXT,
  status           TEXT DEFAULT 'open', -- open | done | expired | cancelled
  created_at       DATETIME,
  closed_at        DATETIME
);

Scheduler Logic

  1. Execute check_command (15s timeout)
  2. Exit 0 → close as done
  3. Exit 2 → skip, advance to next interval
  4. Exit 1 or no command → spawn agent with followup context
  5. Final attempt: agent must resolve, escalate, or cancel
  6. Busy agent guard: retry in 2 minutes without consuming attempt

Agent Context (on followup invocation)

# Follow-Up Check (attempt 2 of 3)
**Description:** PR review pending
**Check result** (exit 1): open
**Previous check output:** open
**Backoff remaining:** 1h
---
[Full system prompt + memory]

Final attempt header includes [FINAL] — agent must make terminal decision.

Server Startup

  1. Create HiveContext (all stores)
  2. Initialize EventBus
  3. Start daemon in-process if no external daemon running
  4. Wire event bus: messages.send() → message:new, state.updateStatus() → agent:state, audit.logInvocation() → audit:invocation
  5. Express app + SSE endpoint + static React build
  6. Hot-reload every 30s (detects new agents)

Source Files Indexed

  1. MEMORY.md — Curated long-term notes (agent-editable)
  2. memory/*.md — Last 3 days of daily logs (auto-appended)

Indexing

  • Hash-based change detection (skip unchanged files)
  • Split by paragraph or fixed size
  • Embed via HuggingFace transformers (local, offline)
  • Runs on daemon startup (background, non-blocking)

Daily Log Format

- [2026-03-28T19:30:00Z] @sender in conversation: message snippet...

Written during checkWork when messages classified as NOTE/QUEUE. Written before marking read (crash safety).
