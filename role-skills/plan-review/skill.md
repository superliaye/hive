# Plan Review

How you evaluate proposals, scaling requests, and strategic decisions before committing resources.

## When to Use

Apply this framework when:
- A department head or agent proposes a new initiative
- A scaling request arrives (new agent, new department)
- You're translating a board directive into org-level action
- You're evaluating competing priorities for resource allocation

Do NOT use for routine task delegation — this is for decisions that commit significant org resources or change direction.

## Step 1: Premise Challenge

Before evaluating the plan itself, challenge the premise:

- **Is this the right problem?** The proposal may solve a real problem, but is it the most important one right now?
- **Is this the right time?** Would waiting 1 week change the calculus? Is there urgency or just momentum?
- **Is this already solved?** Check if another agent, an existing priority, or a different approach already addresses this.

If the premise fails, reject or redirect. Don't optimize a plan for the wrong problem.

## Step 2: Scope Mode

Select the right scope posture based on current org state:

| Mode | When to use | Default for |
|---|---|---|
| **Expand** | Org is under capacity, board is pushing growth, no active fires | New product initiatives |
| **Selective expand** | Org is at capacity but this is strategic | Most scaling requests |
| **Hold** | Org is stretched, active fires, or unclear board direction | Default when unsure |
| **Reduce** | Org is overloaded, quality is dropping, too many active priorities | Crisis response |

When in doubt, default to **Hold**. Expanding is easy to approve and hard to undo. Holding preserves optionality.

## Step 3: Alternatives

Never approve the first approach presented. Require:

- **At least 2 distinct approaches.** Not variations of the same idea — genuinely different strategies. "Build it ourselves" vs "reprioritize existing work" vs "decompose differently."
- **Trade-offs for each.** What does each approach cost in time, focus, and resources? What does each sacrifice?
- **Why the recommended approach wins.** The proposer should have a clear recommendation with reasoning, not just options.

If the proposal only presents one approach, send it back: "What alternatives did you consider?"

## Step 4: Decision Principles

When evaluating trade-offs, apply these in order:

1. **Focus over throughput.** An agent doing one thing well beats an agent doing three things poorly. If the proposal adds cognitive load to an already-stretched agent, that's a cost.
2. **Reversible over irreversible.** Prefer decisions you can undo. Hiring a new agent is more reversible than restructuring a department.
3. **Evidence over intuition.** If the proposal claims "the team is overloaded," where's the evidence? Backlog depth, blocked items, focus signals.
4. **Ship over perfect.** If two approaches are close, pick the one that ships sooner. You can iterate.
5. **Explicit over clever.** Simple plans that everyone understands beat elegant plans that require explanation.

## Step 5: Quality Gate

Before approving, verify:

- [ ] Premise survives challenge (right problem, right time)
- [ ] Scope mode is appropriate for current org state
- [ ] At least 2 alternatives were considered
- [ ] Owner is identified (who executes this?)
- [ ] Success criteria are defined (how do we know it worked?)
- [ ] Rollback plan exists (what if it fails?)

If any box is unchecked, send it back with specific questions. Do not approve incomplete proposals.

## Output

After review, communicate your decision clearly:

**Approved:** "Approved. [Owner] owns this. Success criteria: [X]. Check in after [timeframe]."

**Sent back:** "Not ready. [Specific gaps]. Rework and resubmit."

**Rejected:** "No. [Reason]. Here's what I'd consider instead: [redirect]."

Never leave a proposal in limbo. Every proposal gets a clear yes, no, or rework within the same cycle you review it.
